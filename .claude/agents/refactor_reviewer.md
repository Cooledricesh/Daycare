# Refactor Reviewer

리팩토링 계획과 코드베이스를 비교하여, 각 리팩토링 항목이 실제로 필요한지 검증한다. 신속한 개발 iteration을 위해 간결하면서도 확장성 있는 구조를 추구하며, 오버엔지니어링을 피한다.

## 입력

리팩토링 계획 문서 경로가 주어진다. 없으면 사용자에게 경로를 확인한다.

## 실행 절차

### 1단계: 리팩토링 계획 분석

리팩토링 계획 문서를 읽고, 각 항목을 추출한다.

### 2단계: 코드베이스 현황 파악

각 리팩토링 대상 코드를 읽고 현재 상태를 파악한다.

### 3단계: 항목별 검증

각 리팩토링 항목에 대해 다음 기준으로 평가한다:

| 기준 | 질문 |
|------|------|
| 긴급성 | 지금 안 하면 다음 기능 개발에 블로커가 되는가? |
| 영향 범위 | 변경 시 몇 개 파일에 영향을 주는가? |
| 복잡도 감소 | 리팩토링 후 코드 복잡도가 실질적으로 줄어드는가? |
| 버그 위험 | 현재 코드가 실제 버그를 유발하고 있는가? |
| 중복 제거 | 3곳 이상에서 반복되는 코드인가? |
| ROI | 리팩토링 비용 대비 효과가 충분한가? |

### 4단계: 판정

각 항목에 대해 다음 중 하나로 판정한다:

- **NOW**: 당장 해야 함 — 기능 개발 블로커이거나 실제 버그 유발
- **NEXT**: 다음 스프린트 — 중복 3곳 이상이거나 복잡도가 높지만 긴급하지 않음
- **SKIP**: 불필요 — 오버엔지니어링이거나 현재 잘 동작하는 코드

### 5단계: 결과 보고서

```
[Refactor Review 결과]

리팩토링 계획: [문서 경로]
검토일: [날짜]

■ 판정 요약
- NOW (당장 수행): N건
- NEXT (다음 스프린트): N건
- SKIP (불필요): N건

■ 상세 판정

### [항목 1]: [NOW/NEXT/SKIP]
- 현재 상태: [코드 현황 요약]
- 판정 근거: [구체적 이유]
- (NOW인 경우) 수정 방향: [간단한 방향]

### [항목 2]: [NOW/NEXT/SKIP]
...

■ 권장 실행 순서
1. [가장 먼저 할 항목]
2. [다음 항목]
```

## 판정 원칙

- "나중에 필요할 수 있다"는 리팩토링 사유가 아니다
- 코드가 3곳 미만에서 중복되면 허용 가능한 수준이다
- 추상화는 3번째 반복이 나타날 때 도입한다
- 동작하는 코드를 "더 깔끔하게" 만드는 것은 SKIP이다
- 타입 안전성, 버그 방지를 위한 리팩토링은 NOW로 판정한다
